# Aiment 開発規約

**Notionページ**: https://www.notion.so/30309a68050b804fa05aeb66f7929d88  
**場所**: 300万 > Dev > aiment_DevBase > 開発規約  
**最終更新**: 2026-02-11

---

## 0. この規約の目的

- 2人で作っても、コードが壊れにくくなるようにする
- 「終わったつもり」のズレを減らす
- 将来見返したときに、変更の理由がわかる状態を保つ

---

## 1. 開発の進め方

1. 週の最初に、やることを2〜3個決める
2. 1つずつ作業し、作業単位ごとに共有する
3. 動作確認してから、PR（変更提案）を作る
4. 相手がレビュー（確認）して問題なければ main に取り込む（マージ）

**補足**:
- **PR（Pull Request）**: 「この変更を main に入れてよいか」を確認してもらう仕組み
- **マージ**: 作業ブランチの変更を main に反映すること

---

## 2. ブランチ運用

| ブランチ | 用途 |
|---|---|
| `main` | 安定版。直接作業しない |
| `feature/...` | 新機能の作業用 |
| `fix/...` | バグ修正の作業用 |

**命名例**:
- `feature/add-room-ui`
- `fix/socket-reconnect`

**基本ルール**:
- 作業は必ず main から新しいブランチを作って始める
- main への直接コミットは禁止
- 取り込みはPR経由のみ

---

## 3. Git操作の基本手順

```bash
# 1. main を最新にする
git switch main
git pull origin main

# 2. 作業ブランチを作る
git switch -c feature/add-room-ui

# 3. 変更を保存する（コミット）
git add .
git commit -m "feat: ルーム参加フォームを追加"

# 4. リモートに送る（push）
git push -u origin feature/add-room-ui

# 5. GitHubでPRを作成し、相手にレビュー依頼
# 6. レビューOK後に main へマージ
# 7. 次の作業前に main を最新化
git switch main
git pull origin main
```

**補足**: `git fetch origin` はリモートの最新情報だけ取得（作業中に安全に状況確認したいときに使う）

---

## 4. コミットメッセージ規約

**形式**: `種類: 変更内容`

| 種類 | 意味 |
|---|---|
| `feat` | 新機能追加 |
| `fix` | バグ修正 |
| `refactor` | 挙動を変えない改善・整理 |
| `docs` | ドキュメント変更 |
| `test` | テスト追加・修正 |
| `chore` | 雑務（依存更新など） |
| `build` | ビルド/依存関係/パッケージ設定 |
| `ci` | CI設定変更 |

**例**:
- `feat: ルーム参加フォームを追加`
- `fix: ルームIDが空のときのエラーを修正`
- `refactor: socket処理を関数に分割`
- `docs: セットアップ手順を更新`

---

## 5. レビュー時の確認ポイント（最小）

- バグが入りそうな処理はないか
- 名前や構成が読みやすいか
- 既存の機能を壊していないか
- 確認手順がPRに書かれているか

---

## 6. SOLID（設計をよくする考え方）

コードを「読みやすく、変更しやすく」するための5つの考え方。

| 文字 | 原則 | 意味 |
|---|---|---|
| S | 単一責任 | 1つのファイル・関数に役割を詰め込みすぎない |
| O | 開放閉鎖 | 既存コードを壊さず、追加で対応しやすくする |
| L | リスコフ置換 | 入れ替えても同じように使える設計にする |
| I | インターフェース分離 | 必要ない機能まで持つ型を作らない |
| D | 依存性逆転 | 具体実装に強く結びつけすぎない |

**運用ルール（初心者向け）**: 毎PRで、どれか1つを意識して1行書く
- 例: `SOLID: S（入力チェックを別関数に分けて役割を分離）`

---

## 7. 開発コマンド（pnpm）

| コマンド | 用途 |
|---|---|
| `pnpm dev` | 普段の開発（まずこれ） |
| `pnpm dev:web` | Web画面だけ起動（UIの確認） |
| `pnpm dev:signaling` | シグナリングサーバーだけ起動（通信まわりの確認） |
| `pnpm lint` | リポジトリ内全アプリにlintを実行（PR前に実行） |
| `pnpm --filter ./apps/web lint` | Web画面（apps/web）だけlint |
| `pnpm --filter @repo/signaling lint` | シグナリングサーバーだけlint |

---

## 8. 完了の基準（DoD: Definition of Done）

「何を満たしたら完了か」をそろえるための基準。人によって完了の判断がズレるのを防ぐ。

**最小DoD**:
1. 動作確認ができている
2. PRで相手の確認が終わっている
3. main に取り込んだ後も動く
