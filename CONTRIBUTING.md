# CONTRIBUTING

このドキュメントは、2人チームで安全に速く開発を進めるための初版ルールです。  
最初はシンプルに運用し、困った時にだけルールを追加してください。

## 1. 開発の進め方（順番）

毎週の最初に、次の順で合意します。

1. 今週のゴールを1-3個決める（MVP優先）
2. ゴールをIssueに分解する（1 Issue = 半日〜1日目安）
3. 担当を分ける（`web` / `signaling` / `shared`）
4. 実装は小さいPRで出す（1 PR = 300行前後まで）
5. レビュー後に `main` へマージ
6. 週末に「振り返り」を10分だけやる（うまくいった点、改善1つ）

## 2. ブランチ運用

- `main`: 常に動く状態を保つ
- 作業ブランチ: `feature/<issue-id>-<short-name>` または `fix/<issue-id>-<short-name>`
- 直接 `main` にpushしない

例:

- `feature/12-add-room-ui`
- `fix/34-socket-reconnect`

## 3. コミットメッセージ規約（Conventional Commits）

形式:

`<type>(optional-scope): <summary>`

例:

- `feat(web): add room join form`
- `fix(signaling): handle missing roomId`
- `refactor(shared): split validation helpers`
- `docs: update setup steps`

使う `type` は最初はこの8個だけに固定します。

- `feat`: 機能追加
- `fix`: バグ修正
- `refactor`: 挙動を変えない改善
- `docs`: ドキュメント変更
- `test`: テスト追加・修正
- `chore`: 雑務（依存更新など）
- `build`: ビルド/依存関係/パッケージ設定
- `ci`: CI設定変更

迷ったら:

- ユーザー価値が増える変更 -> `feat`
- バグを直す変更 -> `fix`
- ふるまい不変の整理 -> `refactor`

## 4. PRルール

- PRタイトルもConventional Commitsに合わせる
- PR本文に次を必ず書く
  - 何を変えたか
  - なぜ必要か
  - どうやって確認したか
- スクリーンショットはUI変更時に添付
- セルフレビューしてからレビュー依頼

## 5. レビュー観点（最低チェック）

レビューは「動くか」より「将来困らないか」を見る。

1. バグの可能性（null/undefined、非同期、例外）
2. 可読性（責務分離、命名、重複）
3. 影響範囲（既存機能を壊していないか）
4. テストや手動確認の妥当性
5. セキュリティ（入力値を信用しない）

## 6. SOLIDを学びながら適用するルール

各PRで、最低1つだけSOLID観点を意識して説明する。

- `S` 単一責任: 1ファイル1責務を意識。肥大化したら分割。
- `O` 開放閉鎖: if分岐追加より、拡張しやすい関数分割を優先。
- `L` リスコフ置換: 置き換え可能な抽象（interface/型）を壊さない。
- `I` インターフェース分離: 使わないメソッドを持つ型を作らない。
- `D` 依存性逆転: 具体実装に直接依存しすぎない（注入可能にする）。

PR本文に1行追加:

`SOLID: S を意識し、socket接続管理と入力検証を別モジュールに分離`

## 7. このリポジトリの開発コマンド

ルートで実行:

- `pnpm dev`: web/signaling を同時起動
- `pnpm dev:web`: webのみ起動
- `pnpm dev:signaling`: signalingのみ起動

追加推奨（今後入れる）:

- lint と test の共通スクリプトをルートに追加
- PR前に lint/test を必須化

## 8. Definition of Done（完了条件）

Issueを完了にする条件:

1. 仕様を満たす
2. 主要動作を確認済み
3. PRレビューで承認済み
4. 必要なドキュメント更新済み
5. リグレッションがない

## 9. トラブル時のルール

- 30分詰まったら、必ずIssue/チャットで共有
- 「試したこと」「エラー内容」「次に試すこと」を書く
- 個人で抱え込まない

## 10. まず最初の1週間でやること

1. このルールで運用開始
2. 1週間後に「不要なルール」を削る
3. 足りない項目だけ追加

このファイルは育てる前提です。厳密さより、継続運用を優先してください。
